# Quickstart

Get from zero to a real, runnable Godot 4 game generated by a local model. This reflects
what works **today** (Phase 0). For the bigger picture see `README.md` and `WHY.md`.

---

## 1. Prerequisites

You need three things on your machine:

1. **Python 3.11+** (`python3 --version`).
2. **Godot 4.x** — the standard build. Confirm `godot --version` prints a `4.x` version. If
   `godot` isn't on your `PATH`, set the full path in `config/playsmith.yaml`
   (`engine.godot.binary`). For HTML5 export you also need the export templates
   (Godot editor → *Editor → Manage Export Templates*).
3. **A local model** via [Ollama](https://ollama.com) (easiest):
   ```bash
   ollama pull qwen2.5-coder:7b      # modest hardware
   # or, stronger machine:
   ollama pull qwen3-coder           # larger, agentic-tuned, big context
   ```
   Ollama serves an OpenAI-compatible endpoint at `http://localhost:11434/v1`.
   Any OpenAI-compatible server works (LM Studio, vLLM, llama.cpp, or a cloud key) — just
   change `base_url`/`model` in the config.

> ⚠️ **Context window matters.** Local models need a large context for agentic editing.
> Keep `num_ctx` at 16K–32K (the config default is 16384). The 4K default breaks file editing.

---

## 2. Install

```bash
git clone <your-fork-or-this-repo> playsmith && cd playsmith
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"      # drop [dev] if you don't want ruff/pytest
```

---

## 3. Configure

```bash
cp config/playsmith.example.yaml config/playsmith.yaml
```

Edit `config/playsmith.yaml` to point at your model and (if needed) your Godot binary, and to
choose where generated games are written (`workspace_dir`, default `~/playsmith-games`).
Then check it resolves:

```bash
playsmith config-check
```

---

## 4. Verify the foundations

Run these in order — each proves one pillar works before you generate a game:

```bash
playsmith models          # round-trips a message to your local model
playsmith engine-check    # scaffolds a trivial Godot project and runs it headless
playsmith skills          # lists the installed game-generation skills (2d-platformer)
```

If `playsmith models` fails, your model server isn't reachable. If `engine-check` fails,
Godot isn't found or isn't 4.x.

---

## 5. Make a game

```bash
playsmith new "a 2D platformer where a cat collects fish and avoids spikes" --yes
```

What happens:
1. The prompt is **routed** to the `2d-platformer` skill.
2. A real Godot 4 project is **scaffolded** in your `workspace_dir`.
3. The agent **writes** scenes and GDScript, then **runs** the game headless, **reads the
   logs**, and **fixes** issues until it runs cleanly — the reality loop.
4. It prints where the project is and how to open it.

Drop `--yes` to approve each file write yourself (you'll see a diff for every change).

---

## 6. Play, edit, export

```bash
playsmith run                       # play the most-recent game in a window
godot --editor --path ~/playsmith-games/<your-game>   # open & edit it by hand
playsmith export --target web       # headless HTML5 export to <project>/build/
python -m http.server -d ~/playsmith-games/<your-game>/build   # then open localhost:8000
```

The generated project is **yours** — real, editable Godot source, no lock-in.

## Troubleshooting

- **Model is flaky / gives up mid-build.** Raise `num_ctx` (16K → 32K), use a stronger model
  (e.g. `qwen3-coder`), or temporarily point the config at a cloud model to confirm the
  pipeline, then switch back. See `BUILD_PLAN.md` → "If a local model struggles."
- **Screenshots are blank.** `--headless` uses a dummy renderer that can't capture pixels.
  On a desktop, screenshots work; on a headless server, wrap with `xvfb-run`.
- **Export fails.** Install the HTML5 export templates in the Godot editor.
- **"No generated project found."** Run `playsmith new "..."` first; `run`/`export` default to
  the most recently generated project (or pass `--project <dir>`).
